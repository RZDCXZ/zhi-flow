# Zhi Flow：单用户 AI 聊天与 RAG 学习项目规格

Status: ready-for-agent

## Problem Statement

用户希望通过一个可运行、可观察、可逐步验收的个人知识库 AI 聊天应用，系统学习现代 AI 应用开发，而不是只做一个“能返回答案”的演示。当前需要把 LLM API、多轮对话、流式输出、文档摄取、Embedding、pgvector 检索、Reranker、可靠队列、引用、拒答、调试和评测串成一条清晰的学习路径。

如果一次性搭建完整系统，学习者很难看清每层协议、状态与失败路径；如果过早引入 Agent、LangChain 或复杂权限，又会掩盖确定性 RAG 的核心机制。因此，本项目需要按严格依赖顺序拆成可独立观察和验收的里程碑，每完成一步先展示结果与原理，再由用户决定是否继续。

## Solution

构建一个单用户、私人部署、桌面优先且基本适配移动端的 Web 应用。用户可以显式选择“通用聊天”或“知识库聊天”：通用聊天支持可靠的多轮流式会话；知识库聊天先将追问改写为独立问题，再从指定知识库召回 Chunk、重排、组装证据并生成带引用的回答，证据不足时明确拒答。

系统使用 Next.js、TypeScript、Tailwind CSS 和 shadcn/ui 构建 Web 层，使用 Supabase PostgreSQL、Storage、pgvector 和 Queues 保存业务数据、文件、向量及摄取任务。模型与供应商信息仅由后端配置，通过薄 Provider 接口隔离；第一版固定使用 1024 维的 `BAAI/bge-m3` Embedding 和 `BAAI/bge-reranker-v2-m3` Reranker。文档 Consumer 以持久队列为边界，展示 visibility timeout、幂等、重试和失败归档。

交付采用 23 个顺序里程碑。每个里程碑都明确学习目标、前置依赖、最小实现、运行观察、验证方法、完成标准与暂不处理项。规格获批前不创建应用骨架、不安装依赖、不编写应用代码；实施后也一次只推进一个里程碑。

## User Stories

1. 作为学习者，我想按依赖顺序完成小型里程碑，以便理解每一层新增了什么能力。
2. 作为学习者，我想在每个里程碑结束时先看到运行结果和原理解释，以便决定是否继续。
3. 作为学习者，我想看到每个里程碑明确的完成标准，以免把“看起来能用”误认为完成。
4. 作为学习者，我想知道每个阶段暂不处理什么，以免范围无意膨胀。
5. 作为用户，我想创建、重命名、查看和删除自己的 Conversation，以便管理不同主题的聊天。
6. 作为用户，我想显式选择通用聊天模式，以便在不检索知识库时直接与模型对话。
7. 作为用户，我想显式选择知识库聊天模式和 Knowledge Base，以便回答只基于指定资料。
8. 作为用户，我不想在前端看到模型 ID、供应商或密钥配置，以便界面聚焦产品任务并避免泄密。
9. 作为用户，我想看到助手逐步流式输出，以便尽快获得反馈。
10. 作为用户，我想停止正在生成的回答，以便控制时间、Token 和无用输出。
11. 作为用户，我想重新生成失败、不满意或已取消的回答，以便保留原问题并创建清晰的新尝试。
12. 作为用户，我想在刷新页面后继续看到完整 Message 历史，以便会话不会因断线丢失。
13. 作为用户，我想区分正在生成、已完成、已取消和失败的助手 Message，以便准确理解当前状态。
14. 作为用户，我想在网络断开或模型失败后看到可行动的错误提示，以便安全重试。
15. 作为用户，我想避免一次重复提交产生多条相同 Message，以便历史记录保持一致。
16. 作为用户，我想创建和管理多个 Knowledge Base，以便隔离不同主题的资料。
17. 作为用户，我想上传文本型 PDF、Markdown 和 TXT Document，以便构建个人知识库。
18. 作为用户，我想在上传前看到文件类型、大小和页数限制，以便知道哪些文件可处理。
19. 作为用户，我想看到 Document 从已上传、已排队、处理中到就绪或失败的状态，以便了解摄取进度。
20. 作为用户，我想看到可理解且不泄露敏感信息的摄取失败原因，以便修复文件后重试。
21. 作为用户，我想对失败的 Document 重新排队，以便无需重复创建记录。
22. 作为学习者，我想观察 Producer 如何把摄取任务可靠地放入队列，以便理解异步边界。
23. 作为学习者，我想观察 Consumer 的 visibility timeout、重试和失败归档，以便理解任务可靠性。
24. 作为学习者，我想看到重复队列消息不会产生重复 Chunk，以便理解幂等实现。
25. 作为学习者，我想预览解析后的页面、标题与文本，以便发现解析问题。
26. 作为学习者，我想预览结构优先的 Chunk 及相邻重叠，以便理解切块参数的影响。
27. 作为学习者，我想查看每个 Chunk 的文档名、页码、标题路径、顺序和原文定位，以便验证可追溯性。
28. 作为学习者，我想观察 Embedding 批处理的请求数量、耗时和失败，以便理解向量化成本与限制。
29. 作为学习者，我想直接运行相似度查询并查看分数，以便理解 pgvector 召回。
30. 作为学习者，我想在独立检索实验页调整查询并查看候选 Chunk，以便在生成答案前调试检索。
31. 作为学习者，我想并排比较重排前后的顺序和分数，以便理解 Reranker 的价值。
32. 作为知识库用户，我想让每个关键事实附带 Citation，以便追溯到原文。
33. 作为知识库用户，我想从 Citation 查看文档名、页码或标题路径及原文摘录，以便核验答案。
34. 作为知识库用户，我想在证据不足时收到明确拒答，以便避免把模型猜测当作知识库事实。
35. 作为知识库用户，我想追问上一个问题时系统能理解省略的上下文，以便自然地进行多轮 RAG。
36. 作为学习者，我想看到追问被改写成什么独立问题，以便判断检索失败是否源于改写。
37. 作为长会话用户，我想让系统使用滚动摘要和最近消息控制上下文，以便对话不会无限增长。
38. 作为学习者，我想观察输入、输出 Token 和上下文预算，以便理解成本和截断取舍。
39. 作为学习者，我想展开 RAG Trace，查看改写、召回、重排、最终上下文、耗时与 Token，以便定位失败阶段。
40. 作为学习者，我想让调试信息与最终答案分开呈现，以便正常使用时保持界面简洁。
41. 作为学习者，我想用约 30 条固定 Eval Case 回归检索命中，以便量化检索改动是否退步。
42. 作为学习者，我想评估回答正确性、引用准确性和拒答，以便把生成质量与检索质量分开。
43. 作为学习者，我想查看评测报告中的单条失败和汇总指标，以便把真实失败加入回归集。
44. 作为维护者，我想记录请求关联 ID、阶段耗时、Token 和脱敏错误，以便调试而不记录 API Key。
45. 作为维护者，我想通过后端配置切换 OpenAI-compatible 聊天模型，以便业务代码和前端无需修改。
46. 作为维护者，我想让 Chat、Embedding 和 Reranker 的超时与重试策略可配置，以便适配不同本地和部署环境。
47. 作为维护者，我想让服务端验证所有模型输出、Citation 和文件输入，以便不把模型结构化输出当作业务保证。
48. 作为维护者，我想让私有 Storage 对象只由服务端读取，以便文档不会因客户端直连而公开。
49. 作为维护者，我想分别部署 Web、数据服务和 Consumer，以便理解无服务器请求与长任务的运行边界。
50. 作为维护者，我想拥有本地完整开发与线上部署说明，以便重复启动、验证和排障。
51. 作为移动端用户，我想完成基本的会话、上传和聊天操作，以便临时使用时不被桌面布局阻塞。
52. 作为用户，我想让删除 Document 后其 Chunk 与向量不再被检索，以便知识库内容保持准确。
53. 作为用户，我想让删除 Knowledge Base 前得到影响提示，以便避免误删 Document 和历史绑定。
54. 作为学习者，我想对超时、限流、流中断、损坏文件和重复任务做可靠性实验，以便理解正常路径之外的系统行为。

## Implementation Decisions

- 系统分为五个业务能力：Conversation/Message、Knowledge Base/Document 摄取、Retrieval/RAG、RAG Trace/Eval，以及后端配置与外部 Provider。UI、API 和 Consumer 复用这些能力，不在路由或组件中复制业务规则。
- 第一阶段只实现确定性的两步 RAG：摄取与索引是离线流水线，改写、召回、重排和生成是在线流水线。Agent、Tool Calling 和模型决定控制流都留到后续阶段。
- Web 技术栈固定为 Next.js、TypeScript、Tailwind CSS 与 shadcn/ui；数据服务固定为 Supabase PostgreSQL、Storage、pgvector 与 Queues。最终 Web 部署至 Vercel，Consumer 独立运行，Supabase 托管数据服务。
- 产品不提供注册、登录、邀请、团队、配额或复杂 RLS。第一版按单用户部署，Storage bucket 保持私有，数据库与 Storage 的特权凭据只在服务端使用；线上环境必须有部署层访问保护，不把无认证应用公开暴露。
- 前端只传产品概念与稳定 ID，不传模型名称、Base URL、API Key、采样参数或供应商字段。密钥永不进入源码、客户端、Document、Prompt、RAG Trace 或日志。
- Provider 采用薄接口：Chat Provider 接收规范化消息、生成参数与取消信号并返回流式增量和最终用量；Embedding Provider 接收文本批次并返回顺序一致的 1024 维向量；Reranker Provider 接收查询与候选 Chunk 并返回每个候选的相关性分数。业务层负责超时、重试、校验和持久化。
- Chat Provider 默认面向 OpenAI-compatible 服务，通过后端配置选择 DeepSeek 或其他兼容模型。切换聊天模型只改配置。Embedding 第一版固定为硅基流动的 `BAAI/bge-m3`、维度 1024；不实现模型切换或索引迁移。Reranker 固定为硅基流动的 `BAAI/bge-reranker-v2-m3`。
- 核心数据模型包括：Knowledge Base；Document；Document Chunk；Conversation；Message；RAG Run；Citation。队列任务只保存处理所需的稳定 ID、任务版本和幂等键，不把整份文件内容放入消息。
- Knowledge Base 保存名称和时间戳。删除采用显式确认；删除后关联 Document、Chunk 和向量必须不可检索。第一版可物理级联删除，不实现回收站。
- Document 保存 Knowledge Base、原始文件名、私有 Storage 对象键、MIME、字节数、页数、SHA-256、摄取状态、当前阶段、尝试次数、错误码、脱敏错误摘要和时间戳。状态固定为 `uploaded`、`queued`、`processing`、`ready`、`failed`、`archived`；只有 `ready` Document 可被检索。
- Document Chunk 保存所属 Document/Knowledge Base、顺序、原文、估算 Token 数、页码范围、标题路径、稳定原文定位、Embedding 模型标识、1024 维向量和时间戳。相同 Document 版本与切块版本的重新处理采用替换式事务，避免新旧 Chunk 混用。
- Conversation 保存标题、模式、可空的 Knowledge Base、滚动摘要、摘要覆盖到的 Message 和时间戳。模式固定为 `general` 或 `knowledge_base`；知识库模式必须绑定一个 Knowledge Base，通用模式不得隐式检索。
- Message 保存 Conversation、角色、正文、状态、客户端幂等键、所对应的原始尝试、错误码和时间戳。角色固定为 `user` 或 `assistant`；用户 Message 入库即为 `completed`，助手 Message 状态固定为 `streaming`、`completed`、`cancelled`、`failed`。重新生成创建新的助手 Message，并关联同一用户 Message，不覆盖旧尝试。
- RAG Run 关联用户 Message 与助手 Message，保存独立问题、配置快照、候选及分数、最终上下文 Chunk、阶段耗时、Token 用量和最终结果。Citation 关联 RAG Run、助手 Message 与真实 Chunk，保存显示顺序、文档显示名、页码范围、标题路径、精确摘录和原文定位；摘录必须能在对应 Chunk 原文中验证。
- 上传第一版只接收文本型 PDF、Markdown 和 TXT。默认每个文件最大 20 MiB，PDF 最大 200 页，解析后文本最大 2,000,000 字符，单次最多选择 10 个文件但逐文件建任务。这些上限均由后端配置并在客户端展示；选择依据是学习项目的本地资源、无服务器请求限制和可重复调试时间，而非产品级大规模吞吐。
- 文件类型按扩展名、声明 MIME 与实际内容特征共同校验；拒绝扫描 PDF、加密 PDF、空文件、损坏文件和不支持的编码。解析错误使用稳定错误码，原始异常仅在脱敏服务端日志中保留。
- Producer 在 Document 与 Storage 对象成功落地后写入队列，并以 Document ID、摄取版本和内容哈希形成幂等键。数据库状态与入队失败必须可恢复；重新入队不创建新的 Document。
- Consumer 一次租约一条消息，默认 visibility timeout 15 分钟，单个任务总时限 10 分钟，最多尝试 5 次；外部 API 短暂错误按带抖动退避重试，耗尽后将 Document 标为 `failed` 并把任务归档到失败队列。重复消息、Consumer 崩溃和租约过期都不得产生重复有效 Chunk。
- 线上 Consumer 作为独立 Node.js 进程持续拉取队列；本地用同一进程入口运行。Vercel 只提供一个受服务端密钥保护、限制执行批数的定时消费端点作为教学和兜底路径，不把长文档的完整处理依赖于单次无服务器请求。
- 切块采用结构优先策略：先尊重 Markdown 标题、段落和 PDF 页边界，再按 Token 上限拆分；默认目标约 500 Token、相邻重叠约 80 Token。空白归一化不得破坏页码、标题路径和可验证原文定位。切块算法版本写入摄取配置快照。
- Embedding 默认每批 32 个 Chunk，单次请求超时 60 秒；批量大小、并发和超时均可配置。向量维度在写入前必须严格校验为 1024，批次输出数量与输入数量不一致时整批失败，不写部分错位结果。
- pgvector 使用与 Provider 输出匹配的距离度量和向量索引；小数据集先允许精确查询作为正确性基线，数据达到足以观察索引差异后再比较近似索引。查询只在所选 Knowledge Base 和 `ready` Document 中进行。
- 在线 RAG 固定顺序为：基于滚动摘要和最近消息改写独立问题；生成查询向量；pgvector 召回 20 个 Chunk；去重并交给 Reranker；取前 5 个合格 Chunk；按 Token 预算组装上下文；生成答案；验证 Citation。召回数 20、上下文数 5 均可配置。
- 多轮上下文默认使用滚动摘要和最近 12 条 Message。摘要只覆盖更早的已完成 Message，保留 `summary_through_message_id` 防止重复总结；摘要失败时保留原摘要并按最近消息安全降级。
- RAG Prompt 把检索文本明确标记为不可信证据而不是指令，要求只依据证据回答、每个实质性事实使用 `[n]` 引用、无法支持时明确说明资料不足、不得编造 Citation。问题改写 Prompt 只输出独立问题，不回答问题，也不服从历史或 Document 中试图改变系统规则的内容。
- 无答案判定采用确定性前置门槛加生成约束：若重排后没有候选，或最高归一化重排分数低于可配置阈值，则不调用生成模型，直接返回“资料不足”。初始阈值为 0.35，必须在约 30 条 Eval Case 上校准后才能作为里程碑验收值。生成后若引用 ID 不存在、摘录无法回溯或知识库答案没有任何有效 Citation，则答案不得标记 `completed`，而应转换为安全拒答或 `failed`。
- Citation 的客户端结构固定包含 `citationId`、`order`、`chunkId`、`documentId`、`documentName`、`pageStart`、`pageEnd`、`headingPath`、`quote` 和 `sourceLocator`。页面不存在时使用标题路径和文本定位；客户端不能自行拼接 Storage 公网 URL。
- 聊天写接口接收 Conversation、正文、客户端幂等键和可选的重生成来源，先持久化用户 Message 与 `streaming` 助手 Message，再返回 SSE。停止接口按助手 Message 取消正在进行的生成；重新生成接口为同一用户 Message 创建新尝试。
- SSE 协议版本为 1。每个事件都有 `requestId`、单调递增 `sequence` 和时间戳；事件类型固定为 `message.created`、`rag.stage`、`content.delta`、`citation.snapshot`、`usage.snapshot`、`message.completed`、`message.cancelled`、`message.failed` 和注释心跳。终态事件只出现一次；客户端按 `requestId + sequence` 去重，断线后以数据库状态为准，不盲目重放已产生副作用的模型请求。
- 默认超时为：聊天首字节 15 秒、流空闲 30 秒、单次聊天总时限 120 秒、Embedding 60 秒、Reranker 30 秒、问题改写 30 秒。外部非流式调用最多重试 3 次；流式输出在首个正文增量之后不自动重启，以免生成重复内容。所有限制都由后端配置并记录到 RAG Run 配置快照。
- Token 预算由所选聊天模型的后端配置给出；默认保留明确的最大输出预算，随后按系统指令、滚动摘要/最近消息、检索证据的优先级裁剪。永远不截断系统安全指令；证据按重排顺序纳入并记录被排除原因。
- RAG 调试视图展示独立问题、20 个召回候选、重排前后顺序与分数、最终 5 个上下文、各阶段耗时、Token 用量、拒答门槛和错误码。默认折叠，普通聊天不伪造 RAG 数据；Prompt 全文和密钥不返回客户端。
- Eval Case 使用版本化 JSONL：包含稳定 ID、类别、Conversation 输入、查询、知识库夹具引用、期望 Chunk、必需事实、禁止断言和是否应拒答。Eval Run 输出每条结果、配置快照与汇总；检索指标至少包括 Recall@20 和最终上下文命中率，生成指标至少包括答案正确性、Citation 准确性、拒答准确性与延迟。
- 评测优先使用确定性规则和人工标注；对开放式正确性可增加 LLM-as-judge，但必须保存评审依据并用人工样本校准，不能让单一 Judge 分数成为唯一发布门槛。
- UI 采用左侧 Conversation/Knowledge Base 导航、中间聊天区、右侧 Citation 与 RAG 调试抽屉。窄屏时左右区域改为抽屉；第一版不做语音、图片生成、复杂动画或供应商配置界面。
- 23 个教学里程碑各自对应一个独立实施票据，按编号串行依赖。任何里程碑只实现其最小纵切片；通过验证、展示结果并得到用户继续指令后，才领取下一票据。

## Testing Decisions

- 好测试只断言用户或外部调用者可观察的行为：HTTP 状态、SSE 事件序列、持久化后的业务状态、可检索结果、Citation 可回溯性、队列重试结果和用户界面状态；不锁定组件内部结构、私有函数、SQL 实现细节或 Prompt 的逐字文本。
- 仓库当前没有应用代码或测试先例，因此不继承低层测试模式。主要新测试接缝是公开 HTTP/SSE API：通过真实本地 PostgreSQL、Storage 与队列，配合确定性的假 Chat/Embedding/Reranker Provider，覆盖 Conversation、Message、上传、检索、RAG、停止、重生成与失败状态。
- 异步摄取保留第二个最高层接缝：向队列放入真实任务后，调用 Consumer 的“处理一条租约消息”入口，使用文件夹具、真实本地数据库/Storage 和假 Embedding Provider，断言 Document 状态、Chunk、向量、重试、幂等和失败归档。该接缝避免为 Parser、Chunker 和仓储层分别建立大量耦合测试。
- 浏览器层仅保留少量关键冒烟测试：创建 Conversation、看到流式增量、停止生成、刷新恢复、上传 Document、在知识库回答中打开 Citation/RAG Trace。算法组合与错误矩阵主要在 API/Consumer 接缝验证，以控制速度和脆弱性。
- Provider 合约使用共享契约测试，确保真实适配器和假适配器对取消、超时、向量维度、候选顺序、用量和错误分类具有一致语义；不在普通测试中调用付费外部 API。真实供应商仅通过显式的手动 smoke 命令验证。
- 切块与 Citation 定位增加少量纯输入输出测试，因为边界、页码和原文摘录是高风险确定性规则；这些测试以完整 Document 夹具的可观察 Chunk 结果为准，不测试内部递归步骤。
- 每个里程碑先写其验收行为，再实现最小代码；下一里程碑的门槛是当前票据所有自动验证通过、手动运行观察已记录、没有泄露密钥或未解释的终态，并由用户明确决定继续。
- 可靠性矩阵必须覆盖：首字节超时、流中断、用户取消、429/5xx、重复幂等键、队列租约过期、Consumer 崩溃、损坏文件、维度错误、空召回、伪造 Citation 和删除后仍可检索。所有失败都需验证业务终态与可安全重试性。
- 评测不追求第一版绝对分数，而要求建立可重复基线和防回退门槛。初始建议门槛为：固定夹具 Recall@20 不低于 90%，期望 Chunk 进入最终上下文不低于 80%，Citation 指向有效 Chunk 为 100%，应拒答样例准确率不低于 90%；阈值可由首个基线报告调整，但调整必须记录理由。

## Out of Scope

- 第一阶段不实现 Agent、Tool Calling、多 Agent、GraphRAG、MCP 工作流或模型自主控制循环。
- 不使用 LangChain 等框架隐藏第一版的模型协议、检索顺序或状态机；稳定后可另立规格评估。
- 不实现注册、产品内认证、邀请、团队协作、租户、复杂 RLS、SaaS 配额、计费或管理后台。
- 不支持扫描 PDF、OCR、Word、Excel、网页抓取、图片理解、音频、语音或图片生成。
- 不实现中文 Hybrid Search、BM25、Multi-query、知识图谱、微调或 Embedding 模型切换与索引迁移。
- 不实现大规模生产吞吐、跨区域高可用、自动扩缩容优化、复杂缓存、灰度发布或完整监控平台。
- 不把模型生成的结构化输出当作业务校验，不执行 Document 中的指令，不记录或展示 API Key。
- 不在规格确认前创建应用骨架、安装依赖、创建数据库迁移或编写应用代码。

## Further Notes

- 本规格使用 `Knowledge Base`、`Document`、`Document Chunk`、`Conversation`、`Message`、`RAG Run`、`Citation` 和 `Eval Case` 作为统一领域词汇；后续领域文档应沿用这些名称，避免用“文件片段”“聊天记录”等近义词混指不同实体。
- 初始仓库没有应用代码、领域词汇表、ADR 或测试先例。本规格中的接口、状态和测试接缝因此属于首批架构决策；若实施前新增 ADR 与之冲突，应先显式解决冲突。
- 研究依据集中在既有 AI 应用学习路线笔记中；规格只保留产品与工程决策，不复制长篇资料。实施时应优先查阅模型厂商、Supabase、Next.js 和 Vercel 的官方文档，并记录与本规格有关的版本差异。
- 规格与 23 个实施票据共同构成最终计划。规格状态为 `ready-for-agent` 仅表示内容已可实施；交接要求仍然是先由用户审阅确认，再从里程碑 1 开始，并且一次只推进一个里程碑。
